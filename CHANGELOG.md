### 2026-06-22 — T3.19 telemetry breadth: the faction-commissary requisition loop — *drop t319-commissary-telemetry* (Opus loop)
Instrumented the last dark faction-gated economy faucet/sink pair: the **commissary requisition loop** (`engine/commissary.py`). The credit legs already ride `credit_flow` (adjust_credits tags `commissary_purchase` / `commissary_purchase_refund` / `commissary_sellback`), but those scattered ledger rows can't be rejoined offline into a per-item **buy→sellback round trip**, carry no faction or item identity, and never reveal the **rank gate** a purchase cleared — so the requisition economy (is gear priced to move at each rank? does the 50%-sellback round-trip get abused?) is unreconstructable from the ledger alone. This drop adds one fail-open, sample-tunable `commissary_txn` event at the two success seams: `purchase_commissary` emits `phase="buy"` with the cost (the sink) + the rank it cleared; `sell_commissary` emits `phase="sell"` with the refund (the faucet). A single event type tagged by `phase` keeps the offline funnel trivial (count/sum buy vs sell per faction/item). New module-level `_emit_commissary_telemetry()` helper (int-coerces char_id, lower-cases the faction, reads `telemetry.commissary_sample` default 1.0, never raises) — mirrors the proven `_emit_trade_telemetry` / `emit_cp_income` pattern. Emits only on the completed-requisition return: every failure branch (insufficient / unknown / rank-locked / refunded-grant; sell not-owned) stays silent. New `telemetry.commissary_sample` tunable (additive YAML, mirrors `p2p_trade_sample`). No requisition math, no schema, no faucet/sink change (telemetry is observe-only), era-clean; `engine/commissary.py` + `data/tunables.yaml` only. New `tests/test_t319_commissary_telemetry.py` (18 tests driving the real `purchase_commissary`/`sell_commissary` against a recording stub DB: per-phase envelope, ledger-leg match, every no-emit failure path, sample-knob honoured, the load-bearing fail-open-when-emit-raises, helper schema/coercion, + source/YAML pins). 94 tests green (commissary telemetry + commissary + sellback + p2p_trade telemetry + hygiene + smoke foundation).

### 2026-06-22 — T3.21 hardening: same-origin CORS posture (decision #6) — *drop cors-same-origin-posture* (Opus loop)
Closed the last open T3.21 launch-hardening item: the deliberate **same-origin CORS posture** (Brian decision #6 — "document + a one-line assert, no middleware"). The SPA, the `/ws` WebSocket, and the chargen/portal JSON API are all served from the single same-origin aiohttp `Application` in `server/web_client.py`, so a browser's same-origin policy already permits the SPA's own `fetch()`/WebSocket calls and blocks any cross-origin page from reading our responses — exactly the protection an unauthenticated public API wants. Emitting any `Access-Control-Allow-*` (CORS) header would only *relax* that default, so we add none and add no CORS middleware. This drop makes the no-op an **enforced invariant** rather than an implicit one: a documented posture block above `_SECURITY_HEADERS` plus a new `_assert_no_cors_headers(headers)` guard called once at import — if a future change pastes a permissive CORS header into the security-headers dict, the module fails to import (caught at boot / by tests) instead of silently opening the surface. No new middleware, no runtime per-request cost, no schema change, no faucet/sink, era-clean; `server/web_client.py` only. Extended `tests/test_t321_security_headers.py` (+5: default set carries no `Access-Control-*` header, the guard passes on the real default and raises (case-insensitively) on a permissive one, live responses — normal + raised-error — carry no CORS header through the middleware, and the posture is documented in source with no `aiohttp_cors` import). 33 tests green (security-headers + hygiene + smoke foundation).
